# `events.json` Schema Reference

The pipeline↔website contract (deployment plan, Phase 0). One JSON document, regenerated
nightly by the exporter stage (Track A) and published to the site repo root alongside the
ICS tree; the website (Track B) renders `/calendar/`, `/agenda/`, and `/feeds/` from this
file and nothing else.

**Filename.** In the site repo the file is served as **`events.json`**. The pipeline repo
keeps the fixture as `events.fixture.json` only to distinguish it from real exporter
output — Track B should copy it into the site repo **as `events.json`**, so the later
swap to real exported data (integration checkpoint) is a no-code file replacement, not a
fetch-path change.

The companion artifact is [`events.fixture.json`](events.fixture.json) — a realistic
hand-curated instance of this schema. Track B builds against the fixture; Track A's
exporter must produce files that make the fixture's structure true of real data. The
fixture is pretty-printed for review; production output is compact (no whitespace).
Consumers must not depend on formatting.

The fixture is generated by [`scripts/make_events_fixture.py`](scripts/make_events_fixture.py)
(hand-authored content, epochs computed from ET wall-clock so DST is exact) and checked by
[`scripts/validate_events_fixture.py`](scripts/validate_events_fixture.py), which
re-derives every rule below and asserts the fixture's edge-case coverage manifest. Its
structural invariants (referential integrity, key/sort order, all-day midnight boundaries,
description word bounds, slug-vs-ICS-file agreement) are exactly the ones a real exported
`events.json` must also satisfy — Track A can lift that half of the checker as a regression
guard, dropping the fixture-only coverage manifest.

**Schema version: 1.** Additive changes (new nullable fields, new lookup entries) do not
bump the version; renames, removals, or semantic changes do. Consumers should ignore
unknown keys.

## Contract-wide semantics

**Feed parity.** `events.json` shows exactly what the ICS feeds show, shaped for a
website. The exporter applies the same rules as `atlanta_events/feed/ical.py`:

- `event.is_hidden = 1` rows are excluded. The file carries no trace of hidden events.
- The published `name` is `COALESCE(display_name, name)` — the moderation override is
  pre-applied and invisible to consumers.
- Occurrences with `starts_at IS NULL` are excluded.
- Retention window: an occurrence appears while its *effective end* — `ends_at`, else
  `starts_at` + the runtime fallback chain, else `starts_at` + 3 h — is newer than
  `generated_at − 24 h` (`FEED_PAST_GRACE_HOURS`). There is no future cutoff; scrapers
  cap intake at 18 months out. Note the window bounds the *end*, not the *start*: a
  long or multi-day occurrence (an exhibition, a multi-day festival) can have a
  `starts_at` days or months in the past while it's still running. **Do not suppress by
  `starts_at`** (that hides ongoing events); if the site hides past items, key on the
  same effective-end rule. A just-ended timed occurrence lingers at most ~27 h.
- Cancelled occurrences are **included** with `status: "cancelled"` (ICS parity:
  subscribers need the tombstone). The site decides whether to strike through or omit.
- An event appears only if it has ≥ 1 in-window occurrence; `occurrences` is never empty.

**Resolution: what is pre-resolved vs. left to the consumer.** The exporter bakes in
everything that requires DB knowledge; the consumer applies exactly three two-level
fallbacks, all of the shape `occurrence ?? event`:

| Value | Consumer expression |
|---|---|
| Venue of an occurrence | `occ.venue_id ?? event.venue_id` |
| Ticket/info link | `occ.ticket_url ?? event.ticket_url` |
| Long description body | `occ.description ?? event.description_full` |

There is no occurrence-level *short* description. `description_short` is event-level and
therefore identical across an event's occurrences by design; what distinguishes sibling
occurrences (a lecture series' per-session topic, a film festival's per-screening synopsis)
lives in `occurrence.description`. **Recommended list-line text:
`occ.description ?? event.description_short`** — prefer the occurrence's own line when it
has one, so two sessions of the same series don't render as identical rows. Detail views
show the resolved body (`occ.description ?? event.description_full`).

Pre-resolved by the exporter (consumers must NOT re-derive):

- `event.venue_id` already includes the org-default fallback
  (`event.location_id ?? organization.default_location_id`).
- `occurrence.ends_at` already includes the runtime chain
  (`occ.runtime ?? event.runtime ?? category.default_runtime`, minutes after start).
  `runtime_minutes` therefore does not appear in this file.
- `name` already includes the `display_name` override.

**Shape.** Occurrences nest under their event (`events[].occurrences[]`) — the site's
unit is one card per event (a production/run/series), not one row per showing. This
deliberately diverges from the flat, one-`VEVENT`-per-occurrence iCal feed; don't flatten
the JSON back to match the ICS. (For a chronological agenda you *derive* a flat list — see
the cookbook — but the stored shape stays nested.)

**Null discipline.** Every documented key is **always present**. Optional fields carry an
explicit `null` when absent — never key-omitted, never a stand-in like `0` or `""`. So
`o.ends_at === null` is the correct test (not `"ends_at" in o`), and an empty string never
means "absent." Required booleans are real `true`/`false`; required strings are non-empty.

**Timestamps.** All are Unix epoch integers in **seconds** (UTC instants; multiply by
1000 for a JS `Date` — `new Date(starts_at * 1000)`), *not* milliseconds. `timezone` is
always `"America/New_York"`; render with browser `Intl` /
`toLocaleString(..., { timeZone: tz })` using the file's `timezone` value — never the
viewer's local zone, and never hardcoded.

**All-day occurrences** (`is_all_day: true`): `starts_at` is ET midnight of the first
day. `ends_at` is **always present in this file** and is ET midnight of the day *after*
the last day (exclusive end, matching the DB convention). Most all-day source rows have a
NULL end in the DB; the exporter **synthesizes** it as the ET midnight following the last
covered day (for a one-day event, the next day's midnight). Render as dates, not times:
the covered days are `[starts_at, ends_at)`.

> **Do not step by 86400 seconds.** ET calendar days are 23, 24, or 25 hours across DST
> transitions, so `ends_at − starts_at` is *not* always a whole multiple of 86 400 (a
> one-day event is usually 86 400 s, but 82 800 or 90 000 on a spring-forward /
> fall-back day). Enumerate the covered days with **ET calendar-date arithmetic**
> (increment the date, re-resolve midnight) — a `t += 86400` loop drops or duplicates a
> day whenever the span crosses a transition. See the calendar recipe below.

*Parity note (producer):* the JSON's all-day end matches the ICS feed — both use the DB's
**exclusive** convention (`ical.py` emits `DTEND = date(ends_at)` directly for a stored
end, and `start_date + 1 day` when the DB end is NULL). Emit the exclusive end per this
schema.

**Timed occurrences:** `ends_at` may be an explicit scraped end time or derived from the
runtime chain — treat it as approximate (fine for "~2 hr" or calendar block sizing; don't
present derived ends as authoritative "ends at 9:43 PM" copy). `null` means no end time
is knowable (no explicit end, empty runtime chain); render start time only.

**Determinism.** Nightly output is committed to the site repo; stable ordering keeps
diffs reviewable. Key order is fixed as documented. Sorting: `categories` and `orgs` by
name (case-insensitive), `venues` by `id`, `events` by `(first *included* occurrence's
starts_at, id)` (an event's earliest occurrence that survived the retention filter —
excluded occurrences don't affect ordering), `occurrences` within an event by
`(starts_at, id)`.

**Size.** Current data (33 feed-eligible orgs): roughly 160 events / ~1,300 in-window occurrences
≈ 400–500 KB compact, well under 100 KB gzipped (GitHub Pages compresses). At ~200 orgs
expect a few MB raw; still a single-fetch payload. Revisit (e.g. month-sharding) only if
this becomes a measured problem.

## Top level

| Key | Type | Description |
|---|---|---|
| `_readme` | array of strings | Human-facing "comment header" — the invariants no single instance can reveal (epoch units, render zone, null discipline, that the file is post-moderation). Non-data; parsers ignore it. Emitted verbatim in production too, so the fact travels with the file |
| `schema_version` | integer | Contract version; this document describes `1` |
| `generated_at` | integer | Epoch of the nightly run. Must be the **same `now_unix` the ICS builders received** — the exporter runs inside the same nightly invocation or is handed the timestamp; it must not call `time.time()` independently, or JSON and ICS retention windows drift apart |
| `timezone` | string | IANA zone all event times belong to; always `"America/New_York"` |
| `feeds` | object | `{ "all": "all.ics", "free": "free.ics" }` — the two global ICS feeds. All `ics` values in this file are paths relative to the directory `events.json` is served from (the ICS tree is published alongside it); the site resolves them against its own base URL, which keeps the file identical across `github.io/<repo>` vs. custom-domain hosting |
| `categories` | array | Full category taxonomy — see below |
| `orgs` | array | Organizations — see below |
| `venues` | array | Venues referenced by this file's events — see below |
| `events` | array | Events with nested occurrences — see below |

## `categories[]`

Always the **complete** taxonomy (27 rows today), including categories with no current
events — every category has a subscriber-stable ICS file, and the `/feeds/` page lists
them all. Sorted by name, case-insensitive.

| Key | Type | Null? | Description |
|---|---|---|---|
| `id` | integer | No | `event_category.id`; referenced by `events[].category_id` |
| `name` | string | No | Display label, e.g. `"Musical Theatre"` |
| `slug` | string | No | Lowercase-hyphenated (`"musical-theatre"`); matches the ICS filename |
| `ics` | string | No | Relative path, `"categories/<slug>.ics"` |

## `orgs[]`

Every org eligible for a per-org feed (`is_scrape_enabled = 1 AND scraper_module IS NOT
NULL` — 33 today), **plus** any org referenced by an event in this file that isn't in
that set. That second bucket happens when an org is disabled while its events are still
in window — the org went dark and its canonical events haven't aged out (their future
occurrences are typically auto-cancelled by then; see `status`). Such orgs have no
`scraper_module`, so both `slug` and `ics` are `null`. Every `events[].org_id` resolves
here. Sorted by name, case-insensitive.

| Key | Type | Null? | Description |
|---|---|---|---|
| `id` | integer | No | `organization.id` |
| `name` | string | No | Org display name |
| `slug` | string | Yes | The org's `scraper_module` (underscores, e.g. `"alliance_theatre"`); matches the ICS filename. **`null` for a referenced-but-ineligible (disabled) org** — those also have `ics: null`. Eligible orgs always have a non-null slug. Note orgs and categories use different slug styles — always read slugs/paths from this file, never derive them from names |
| `website_url` | string | Yes | Org homepage; may be `null` (some disabled orgs lack one) |
| `ics` | string | Yes | Relative path `"orgs/<slug>.ics"`; `null` when the org has no per-org feed (disabled org). A non-null value is safe to link even if the feed is currently empty |

## `venues[]`

Only venues referenced by this file's events (event- or occurrence-level). Sorted by
`id`.

| Key | Type | Null? | Description |
|---|---|---|---|
| `id` | integer | No | `location.id` |
| `name` | string | No | Venue name, e.g. `"Shakespeare Tavern Playhouse"` |
| `street` | string | Yes | Street address |
| `city` | string | Yes | City (metro area — many distinct cities) |
| `state` | string | No | Always `"GA"` in practice |
| `zip` | string | Yes | Zip as string (may be zip+4) |
| `lat` / `lng` | number | Yes | Decimal coordinates (Google Places enrichment); null when not yet enriched. Both set or both null |

## `events[]`

One entry per production/series/one-off with at least one in-window occurrence.

| Key | Type | Null? | Description |
|---|---|---|---|
| `id` | integer | No | `event.id` — stable across nightly regenerations; safe for anchors/permalinks |
| `org_id` | integer | No | → `orgs[].id` |
| `category_id` | integer | Yes | → `categories[].id`; `null` = not yet categorized (site needs an "uncategorized-safe" render path) |
| `venue_id` | integer | Yes | → `venues[].id`; default venue for occurrences. Already includes the org-default fallback. `null` = no venue known. **Defensive case:** every org currently has a default location, so the live pipeline does not yet emit a null event venue — but the field is nullable, the fixture exercises it (a site-specific walk), and the site must render "Venue TBA" rather than crash |
| `name` | string | No | Published title (`display_name` override pre-applied). May contain arbitrary punctuation/Unicode — escape for HTML |
| `subtitle` | string | Yes | Secondary title ("Season 50 Return Engagement"); rarely populated |
| `description_short` | string | Yes | AI-written one-liner (8–22 words) — agenda/list line |
| `description_full` | string | Yes | AI-written body (10–120 words, plain text, no markup) — detail view. `short`/`full` are either both present or both null; null = describer hasn't produced verified copy — render name-only, no placeholder text |
| `ticket_url` | string | Yes | Event-level ticket/info link; may carry long query strings — escape attributes. May be null even when occurrences also have no link: render without a link |
| `occurrences` | array | No | Never empty; sorted `(starts_at, id)` |

### `events[].occurrences[]`

| Key | Type | Null? | Description |
|---|---|---|---|
| `id` | integer | No | `occurrence.id`. Also the ICS identity: UID is `occ-<id>@atlantaeventcalendar.dev` (the current `FEED_UID_DOMAIN`) — useful for deep-linking/correlation with feeds |
| `starts_at` | integer | No | Epoch start (never null in this file) |
| `ends_at` | integer | Yes | Epoch end — see contract-wide semantics (all-day: always present, exclusive; timed: possibly derived, possibly null) |
| `venue_id` | integer | Yes | Override; `null` = inherit `event.venue_id` |
| `subvenue` | string | Yes | Named space within the venue ("Hertz Stage", "Lumiere Lounge") |
| `is_all_day` | boolean | No | Date-scoped rather than time-scoped |
| `is_free` | boolean | No | Free vs. paid (`false` = paid or unknown) — powers the free filter; the site's free view should match `free.ics` |
| `status` | string | No | `"scheduled"` or `"cancelled"` — nothing else |
| `ticket_url` | string | Yes | Override; `null` = inherit `event.ticket_url` |
| `description` | string | Yes | Occurrence-specific body (film-festival synopses, lecture-series topics). Replaces `description_full` in detail views for this occurrence when present, and is the recommended list-line text too (see Resolution above) — it's what distinguishes sibling occurrences |

## Intentionally absent

- **Hidden events, `display_name`, `is_hidden`** — moderation is pre-applied upstream.
- **`runtime_minutes`** — folded into `ends_at`.
- **`sequence`, `updated_at`, `last_seen_at`** — ICS-subscriber machinery; a static site
  re-renders from scratch each visit.
- **Raw/staging data, fetch health, reconciliation issues** — pipeline-internal.
- **Org descriptions** — not needed by the current three pages; additive later if an
  `/about/` page wants them.

## Consumer cookbook

- **Reference data (do this once):** build `id → row` maps for `orgs`, `venues`, and
  `categories`. An event carries `org_id`, `venue_id`, `category_id` (and occurrences carry
  their own `venue_id`); resolve each through its map to get name / slug / address /
  coordinates. Category is intentionally a normalized reference like org and venue — the
  event holds `category_id`, and the label + kebab `slug` live in `categories[]` (which the
  `/feeds/` page needs in full anyway). **Never render a raw `*_id`.**
- **ET day helpers (define once).** Both must operate in ET regardless of the viewer's or
  build runner's timezone — the two traps below hide here.
  - `const dayKey = ep => new Date(ep*1000).toLocaleString("en-CA", { timeZone: data.timezone }).slice(0,10)`
    → the ET `YYYY-MM-DD` an epoch-seconds instant falls on. Pins the zone, so it's correct
    for a viewer in California or a CI runner in Kolkata.
  - `const nextDay = key => { const [y,m,d] = key.split("-").map(Number); return new Date(Date.UTC(y, m-1, d+1)).toISOString().slice(0,10); }`
    → the next calendar day of a `YYYY-MM-DD` **string**. Uses `Date.UTC` so the arithmetic
    can't skew across DST or the local zone.
  - All-day occurrence → the ET days it covers, `[starts_at, ends_at)`:
    `for (let k = dayKey(o.starts_at); k < nextExclusive; k = nextDay(k)) days.push(k)` where
    `nextExclusive = dayKey(o.ends_at)` (string compare works — dates are lexically ordered).
- **Calendar page:** bucket each occurrence into the cell(s) it touches. Timed occurrence →
  `dayKey(o.starts_at)`. All-day occurrence → every ET day from the helper above.
  **Never enumerate by stepping a `Date` (`setDate`/`+86400`) and keying through `dayKey`** —
  a `Date` advances in the *local* zone while `dayKey` reads ET, so they disagree across the
  ET-midnight boundary and drop or duplicate a day (whenever the viewer/runner isn't in ET,
  or across the Nov 1 DST fall-back). Iterate ET day-strings.
- **Agenda page:** flatten `events.flatMap(e => e.occurrences.map(o => ({e, o})))`.
  For a day-grouped agenda, place each `{e,o}` under `dayKey(o.starts_at)`, **and repeat
  all-day occurrences under every ET day they cover** (same helper as the calendar) so a
  multi-day exhibition shows up on Saturday for someone browsing Saturday — don't file it
  only under its start day. Sort within a day by `starts_at` (all-day first). Line text:
  `o.description ?? e.description_short`; resolve venue/link via the fallbacks.
- **Free filter:** `o.is_free === true`. The site's free view should match `free.ics`.
- **Feeds page:** `feeds.all`, `feeds.free`, then `categories[]` verbatim and the `orgs[]`
  **where `ics` is non-null** (skip disabled orgs — no feed to subscribe to). Names,
  slugs, and paths all come from the file. Per-feed event counts, if wanted, are derivable
  by counting occurrences per org/category (decide whether to include cancelled ones).
- **Escaping:** names, descriptions, and URLs are untrusted scraped text — HTML-escape
  everything; attribute-escape URLs.

## What the fixture exercises

`events.fixture.json`: `generated_at` = Mon 2026-07-06 03:12 ET; 20 events, 68
occurrences, 14 venues; all 27 categories, all 33 eligible orgs **plus** 1 disabled org
(34 in `orgs[]`). Deliberate cases —

- **Time-of-day / window:** in-grace **past** occurrence (Sun Jul 5 19:30, inside the
  24 h grace); a "tonight" occurrence on the generation date; a 10:30 PM late show;
  same-day matinee + evening pairs; a timed run crossing the **DST fall-back**
  (Sun 2026-11-01 14:30 matinee, EST); a far-future single occurrence (**Dec 2027**,
  ~17 months out, near the ~18-month scrape horizon) for full-span calendar navigation.
- **Cancellation:** one **cancelled** performance mid-run (otherwise-live event); one
  **wholly cancelled** event (the disabled org's, below).
- **All-day:** single-day (free, subvenue, next-midnight synthesized end); **multi-day**
  (3-day makers' market, exclusive end); a **multi-week span crossing the DST fall-back**
  (an exhibition covering Oct 20–Nov 14, i.e. `ends_at` = the exclusive Nov 15 midnight —
  its `ends_at − starts_at` is deliberately *not* a multiple of 86 400, so a naive stepping
  loop is caught); an **all-day paid** admission
  as well as all-day free.
- **`ends_at` variants:** explicit scraped end (film); category-default derived;
  event-runtime derived; **null** (no chain at all); synthesized all-day ends.
- **Retention edge:** an **ongoing long exhibition** that opened three weeks *before*
  `generated_at` and is still running — `starts_at` well in the past, kept in-window by
  its future effective end (the "don't suppress by `starts_at`" case).
- **Venue variants:** occurrence-level **override** (benefit's second night across town);
  **null-venue** event (site-specific walk → "Venue TBA"); an **unenriched venue** with
  null `street`/`zip`/`lat`/`lng`; subvenues.
- **Org/feed variants:** a **disabled org** (real City Springs Theatre — `slug: null`,
  `ics: null`, `website_url: null`) still referenced by an in-window (cancelled) event,
  exercising the `orgs[]` union rule and the feeds-page skip.
- **Data-thinness:** event with **no descriptions at all**; event with **no ticket URL at
  any level**; **uncategorized** event (`category_id: null`).
- **Per-occurrence** descriptions and ticket URLs (a jazz-talk series with distinct
  per-session topics/links → distinct agenda lines).
- **Escaping:** punctuation-heavy title (`NieR: Orchestra Concert 12026 [YoRHa]!`),
  quotes inside names, long query-string ticket URLs.
